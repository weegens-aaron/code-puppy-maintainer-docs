Feature: Round-Robin Model Distribution

What Does This Do?

Round-robin model distribution allows Code Puppy to automatically cycle through multiple AI models when processing requests, distributing the load across different models to overcome rate limits and improve reliability.

Why Does This Exist?

  • Rate Limit Mitigation: When a single model hits API rate limits, requests can be distributed across multiple models
  • Load Balancing: Spreads computational load across different model instances
  • Cost Distribution: Can mix different pricing tiers (e.g., cheaper models for simple tasks, expensive for complex)
  • Redundancy: If one model is temporarily unavailable, others can still serve requests

How It Works

User Perspective

Users configure a "round_robin" model in their configuration file, specifying multiple candidate models. When Code Puppy uses this model, it automatically cycles through them in sequence.

System Perspective

The RoundRobinModel class wraps multiple model instances and maintains a request counter. After every rotate_every requests, it advances to the next model in the list.

flowchart TD
    A[Request to round_robin model] --> B{Request count >= rotate_every?}
    B -->|No| C[Use current model]
    B -->|Yes| D[Advance to next model]
    D --> E[Reset counter]
    C --> F[Send request]
    E --> F
    F --> G[Return response]

Configuration

Basic Round-Robin

{
  "name": "my_round_robin",
  "model_type": "round_robin",
  "models": ["model_a", "model_b", "model_c"]
}

With Rotation Control

{
  "name": "balanced_rr",
  "model_type": "round_robin",
  "models": ["claude-3.5", "gpt-4o"],
  "rotate_every": 2
}
Setting Purpose Default
models List of model names to cycle through Required
rotate_every Number of requests before rotating to next model 1

Rotation Behavior

Default (rotate_every: 1)

Request 1 → Model A
Request 2 → Model B
Request 3 → Model C
Request 4 → Model A (wraps around)

Custom (rotate_every: 3)

Request 1 → Model A
Request 2 → Model A
Request 3 → Model A
Request 4 → Model B
Request 5 → Model B
Request 6 → Model B
Request 7 → Model C

Thread Safety

The implementation uses a threading.Lock to ensure thread-safe rotation when multiple requests are processed concurrently.

Error Handling

WARNING

Unlike fallback models, round-robin does not retry with other models on error. If the selected model fails, the error is propagated to the caller.

Model Name Display

The round-robin model displays a composite name showing all candidates:

round_robin:model_a,model_b,model_c

With custom rotation:

round_robin:model_a,model_b:rotate_every=3

Testing This Feature

  1. Configure a round-robin model in your config
  2. Run multiple tasks sequentially
  3. Observe which model is used for each request (check logs or model_name property)
