Feature: JSON Agents

What Does This Do?

JSON Agents allow you to define custom AI agents using simple JSON configuration files instead of writing Python code. This enables rapid prototyping, sharing agent configurations, and creating specialized agents without touching the codebase.

Why Does This Exist?

  • Accessibility: Non-programmers can create custom agents
  • Portability: JSON configs are easy to share and version control
  • Rapid Iteration: Edit agent behavior without restarting the application
  • Specialization: Create domain-specific agents (e.g., "code-reviewer", "docs-writer")

How It Works

User Perspective

  1. Create a JSON file in ~/.code_puppy/agents/ or .code_puppy/agents/
  2. Define agent properties: name, description, system prompt, tools
  3. The agent appears in the agent picker alongside built-in agents
  4. Select and use the agent like any other

System Perspective

flowchart TD
    A[User creates agent.json] --> B[discover_json_agents scans dirs]
    B --> C{Valid JSON?}
    C -->|Yes| D[JSONAgent instantiated]
    C -->|No| E[Skip with warning]
    D --> F[Register in _AGENT_REGISTRY]
    F --> G[Agent appears in picker]
    G --> H[User selects agent]
    H --> I[JSONAgent loaded from path]
    I --> J[Tools & prompt configured]
    J --> K[Agent runs with pydantic-ai]

JSON Schema

{
  "name": "agent-identifier",
  "display_name": "Friendly Name 🤖",
  "description": "What this agent does",
  "system_prompt": "Your system prompt here",
  "user_prompt": "Optional default user prompt",
  "tools": ["tool1", "tool2"],
  "tools_config": {},
  "model": "optional-pinned-model"
}

Required Fields

Field Type Description
name string Unique agent identifier (lowercase, hyphens)
description string Brief description shown in agent picker
system_prompt string \ list Agent's system instruction
tools list List of tool names the agent can use

Optional Fields

Field Type Description
display_name string Human-readable name with emoji
user_prompt string Default user prompt for tasks
tools_config object Tool-specific configuration
model string Pin this agent to a specific model

Tool Configuration

Built-in Tools

{
  "tools": [
    "read_file",
    "edit_file",
    "command_runner",
    "browser_control"
  ]
}

Universal Constructor (UC) Tools

UC tools are identified by prefixing with uc::

{
  "tools": [
    "read_file",
    "uc:my-custom-tool"
  ]
}
NOTE

The system automatically detects UC tools and marks them with uc: prefix internally.

Configuration Examples

Simple Code Reviewer

{
  "name": "code-reviewer",
  "display_name": "Code Reviewer 👀",
  "description": "Reviews code for bugs and style issues",
  "system_prompt": "You are an expert code reviewer. Focus on:\n1. Bug detection\n2. Code style\n3. Security issues\n4. Performance",
  "tools": ["read_file", "command_runner"],
  "model": "claude-3-5-sonnet"
}

Multi-Prompt Agent

{
  "name": "docs-writer",
  "display_name": "Documentation Writer 📝",
  "description": "Writes and updates documentation",
  "system_prompt": [
    "You are a technical writer specializing in documentation.",
    "Your goal is to create clear, comprehensive docs.",
    "Always follow the project's style guide."
  ],
  "user_prompt": "Please write documentation for the following feature:",
  "tools": ["read_file", "edit_file", "command_runner"]
}

Discovery Locations

Agents are discovered in two directories (in priority order):

Priority Location Purpose
1 .code_puppy/agents/ Project-specific agents
2 ~/.code_puppy/agents/ User-wide agents
WARNING

Project agents take precedence over user agents when names collide.

Agent Lifecycle

sequenceDiagram
    participant User
    participant CLI
    participant AgentMgr
    participant JSONAgent
    participant Registry

    User->>CLI: /agent list
    CLI->>AgentMgr: get_available_agents()
    AgentMgr->>AgentMgr: discover_json_agents()
    AgentMgr->>JSONAgent: Load each .json file
    JSONAgent-->>AgentMgr: Validated config
    AgentMgr->>Registry: Register agent path
    Registry-->>CLI: Agent list
    CLI-->>User: Display agents

    User->>CLI: /agent switch my-agent
    CLI->>AgentMgr: get_agent_by_name("my-agent")
    AgentMgr->>JSONAgent: JSONAgent(path)
    JSONAgent-->>AgentMgr: Instantiated agent
    AgentMgr-->>CLI: Agent instance
    CLI-->>User: Switched to my-agent

Edge Cases & Gotchas

WARNING

Name Collisions: If a Python agent and JSON agent share the same name, the Python agent takes precedence.

NOTE

Hot Reloading: JSON agents support refresh_config() - changes to JSON files are picked up without restart when using /config reload-agent.

CAUTION

Invalid JSON: Malformed JSON files are silently skipped with a debug log entry.

Error Scenarios

Scenario Behavior User Sees
Missing required field Agent skipped during discovery Warning in agent list
Invalid JSON syntax Agent skipped Warning in agent list
Unknown tool in config Tool filtered out Agent loads without that tool
File not found Agent skipped Warning in agent list

Testing This Feature

Create a Test Agent

mkdir -p ~/.code_puppy/agents
cat > ~/.code_puppy/agents/test-agent.json << 'EOF'
{
  "name": "test-agent",
  "display_name": "Test Agent 🧪",
  "description": "A test agent for verification",
  "system_prompt": "You are a test agent.",
  "tools": ["command_runner"]
}
EOF

Verify Discovery

# Run code_puppy and check agent list
# Or use the API:
curl http://localhost:8000/api/agents

Test Hot Reload

# Edit the JSON file
# Then in code_puppy:
/config reload-agent test-agent
