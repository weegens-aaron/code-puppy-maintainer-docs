Flow: Stream Event Processing

What Happens?

Stream event processing is the real-time event pipeline that captures and broadcasts streaming events from agent runs to registered plugins. As an agent generates tokens, thinks, or calls tools, each streaming event is captured, enriched with metadata, and dispatched to any plugins that have registered stream_event callbacks. This enables features like frontend WebSocket updates, real-time metrics tracking, and external monitoring.

Trigger

What initiates this flow?

  • Agent run starts (main agent or sub-agent)
  • Agent begins generating streaming responses via pydantic-ai event stream
  • Events are emitted for each part (text, thinking, tool calls) and delta

Event Types:

  • part_start - A new part begins (text, thinking, or tool call)
  • part_delta - Content is being streamed (token by token)
  • part_end - A part completes

Outcome

When this flow completes successfully:

  • All streaming events are captured and broadcast to registered callbacks
  • Plugins can react in real-time (e.g., frontend updates, logging, metrics)
  • Event data is sanitized to prevent exposure of sensitive information
  • Errors in callbacks don't break the streaming pipeline

Step-by-Step

    D --> G[Fire stream_event Callback]

    H -->|No| J[Skip Callbacks]

1. Event Stream Initialization

  • What: Agent run begins, event stream is created by pydantic-ai
  • Where: code_puppy/agents/event_stream_handler.py (main) or subagent_stream_handler.py (sub-agent)
  • Failure mode: If stream handler fails, agent run may fail or produce no output

2. Event Capture (part_start)

  • What: When a new part begins (text, thinking, or tool call), capture metadata
  • Where: event_stream_handler.py line ~140-170
  • Data captured:
  • index: Part index in the stream
  • part_type: Type of part (e.g., "TextPart", "ToolCallPart")
  • part: The full part object
  • Failure mode: Missing metadata may cause plugins to misidentify event type

3. Event Capture (part_delta)

  • What: As content streams in, capture incremental updates
  • Where: event_stream_handler.py line ~175-240
  • Data captured:
  • index: Part index
  • delta_type: Type of delta (e.g., "TextPartDelta", "ToolCallPartDelta")
  • delta: The delta object with content changes
  • Token tracking: For text/tool parts, estimate tokens from content length
  • Failure mode: Token estimates may be inaccurate (4 chars ≈ 1 token heuristic)

4. Event Capture (part_end)

  • What: When a part completes, capture final state
  • Where: event_stream_handler.py line ~245-280
  • Data captured:
  • index: Part index
  • next_part_kind: Kind of next part (if available)
  • Cleanup: Reset tracking state (token counts, tool names, etc.)
  • Failure mode: State leaks if cleanup fails (rare)

5. Callback Dispatch

  • What: Fire stream_event callback for each event
  • Where: code_puppy/callbacks.py on_stream_event() function
  • How: Uses asyncio.create_task() to run callbacks non-blocking
  • Session context: Includes optional agent_session_id for tracking
  • Failure mode: Callback errors are logged but don't break streaming

6. Plugin Processing

  • What: Registered plugins receive and process events
  • Where: Plugin-specific handlers (e.g., frontend_emitter/register_callbacks.py)
  • Common actions:
  • Sanitize event data (truncate strings, limit dict sizes)
  • Emit to WebSocket/frontend
  • Log metrics or track tool calls
  • Failure mode: Plugin errors are caught and logged

Data Transformations

Stage Input Output Location
Event Capture PartStartEvent, PartDeltaEvent, PartEndEvent Dict with event_type, event_data event_stream_handler.py
Callback Dispatch event_type, event_data, agent_session_id List of callback results callbacks.py
Plugin Sanitization Raw event data Sanitized dict (truncated strings, limited size) frontend_emitter/register_callbacks.py
Frontend Emission Sanitized event WebSocket message frontend_emitter/emitter.py

Failure Handling

Failure Point Detection Recovery Alert?
Callback dispatch Exception in on_stream_event() Log error, continue streaming No
Plugin processing Exception in plugin callback Log error, continue No
Event loop unavailable RuntimeError (no running loop) Skip callback, log debug No
Sanitization error Exception in _sanitize_event_data() Log error, use fallback No
WARNING

Stream event callbacks run asynchronously and non-blocking. If a callback takes too long, it won't block the agent stream, but it may cause memory buildup if many events pile up.

Performance Characteristics

  • Typical duration: Callbacks execute in <10ms each
  • Throughput: Can handle 100s of events per second
  • Bottlenecks:
  • Slow plugin callbacks (should be async and non-blocking)
  • Large event data (sanitization helps)
  • Scaling considerations:
  • Callbacks run in asyncio.create_task() - don't block
  • Use asyncio.gather() patterns in plugins for parallelism

Debugging This Flow

Key Log Messages

Message Meaning Action
Emitted stream_event: {event_type} Event successfully emitted by frontend_emitter Normal operation
Failed to emit stream_event: {error} Plugin failed to emit event Check plugin code
Error firing stream event callback: {error} Callback dispatch failed Check callbacks module
No event loop available for stream event callback Callback skipped (shutdown context) Normal during shutdown

Common Issues

Symptom Likely Cause Fix
Events not reaching frontend Frontend emitter plugin not loaded Check plugin registration
Large payloads in frontend Sanitization not working Check _sanitize_event_data()
Callbacks not firing No stream_event callbacks registered Use register_callback("stream_event", func)
Memory buildup Slow callbacks blocking Ensure callbacks are async and non-blocking

Trace a Stream Event

  1. Enable debug logging: ```bash export LOG_LEVEL=DEBUG code_puppy ```
  1. Watch for event logs: ``` DEBUG: code_puppy.callbacks: Triggering 1 async callbacks for phase 'stream_event' DEBUG: code_puppy.plugins.frontend_emitter.register_callbacks: Emitted stream_event: part_start ```
  1. Check callback registration: ```python from code_puppy.callbacks import count_callbacks print(count_callbacks("stream_event")) # Should be > 0 if plugins loaded ```
