Flow: Adding a Model

What Happens?

This flow allows users to browse available AI models from the models.dev database, select a model, provide necessary credentials, and add it to the local extra_models.json configuration for use with Code Puppy.

Trigger

  • User action: Running /addmodel or /models add command in the CLI
  • Entry point: code_puppy/command_line/add_model_menu.py → interactive_model_picker()

Outcome

  • A new model entry is added to extra_models.json with proper configuration
  • Required API credentials are saved to user config
  • The model becomes available for selection in the model picker

Step-by-Step

flowchart TD
    A[User runs /addmodel] --> B[Initialize ModelsDevRegistry]
    B --> C{API available?}
    C -->|Yes| D[Fetch from models.dev API]
    C -->|No| E[Fallback to bundled models_dev_api.json]
    D --> F[Display Provider List]
    E --> F
    F --> G{User selects provider}
    G --> H[Display Models for Provider]
    H --> I{User selects model}
    I -->|Custom model| J[Prompt for model ID & context size]
    I -->|Listed model| K[Check tool calling support]
    J --> L[Check credentials]
    K --> L
    L --> M{Missing credentials?}
    M -->|Yes| N[Prompt for API keys]
    M -->|No| O[Add to extra_models.json]
    N --> O
    O --> P[Model ready to use]

1. Initialize Registry

  • What: Load the models database from either live API or bundled JSON
  • Where: AddModelMenu._initialize_registry() in add_model_menu.py
  • Failure mode: If both API and bundled file fail, show warning and exit

2. Browse Providers

  • What: Display paginated list of model providers (OpenAI, Anthropic, Groq, etc.)
  • Where: AddModelMenu._render_provider_list()
  • Failure mode: Unsupported providers (e.g., AWS Bedrock, Google Vertex) are marked with ⚠️ and cannot be selected

3. Select Provider

  • What: User navigates with ↑/↓, pages with ←/→, enters provider with Enter
  • Where: Key bindings in AddModelMenu.run()
  • Failure mode: N/A - navigation is always available

4. Browse Models

  • What: Display models for selected provider with capability icons (👁 vision, 🔧 tool calling, 🧠 reasoning)
  • Where: AddModelMenu._render_model_list()
  • Special option: "✨ Custom model..." at end of list for manual entry

5. Select Model

  • What: User selects a model or custom model option
  • Where: AddModelMenu._add_current_model()
  • Failure mode:
  • If unsupported provider: show error and exit
  • If custom model: enter custom model flow
  • If listed model: enter credential check flow

6. Check Tool Calling Support

  • What: Warn user if model doesn't support tool calling (cannot use tools)
  • Where: AddModelMenu._render_model_details() shows ⚠️ warning
  • Failure mode: User can still proceed but gets confirmation prompt

7. Prompt for Credentials

  • What: Check environment for required API keys, prompt for missing ones
  • Where: AddModelMenu._prompt_for_credentials()
  • Failure mode: User can skip credentials (can be set later with /set)

8. Add to Configuration

  • What: Build model config and save to extra_models.json
  • Where: AddModelMenu._add_model_to_extra_config()
  • Failure mode: JSON parse errors, file write errors

Data Transformations

Stage Input Output Location
Provider selection ProviderInfo Stored in self.current_provider _enter_provider()
Model selection ModelInfo Stored in self.pending_model _add_current_model()
Config build ModelInfo + ProviderInfo Model config dict _build_model_config()
Save {model_key: config} Updated extra_models.json _add_model_to_extra_config()

Configuration Format

The resulting extra_models.json entry looks like:

{
  "groq-llama-3.1-70b-versatile": {
    "type": "custom_openai",
    "name": "llama-3.1-70b-versatile",
    "custom_endpoint": {
      "url": "https://api.groq.com/openai/v1",
      "api_key": "$GROQ_API_KEY"
    },
    "context_length": 131072,
    "supported_settings": ["temperature", "seed", "top_p"]
  }
}

Failure Handling

Failure Point Detection Recovery Alert?
No models data Empty providers list Show warning, exit Yes
Unsupported provider provider.id in UNSUPPORTED_PROVIDERS Block selection, show reason Yes
No tool calling not model.tool_call Show warning, confirm with user Yes
Missing credentials os.environ.get(env_var) returns None Prompt user, allow skip No
JSON parse error json.JSONDecodeError Show error, abort Yes
File write error Exception in atomic write Show error, abort Yes

Performance Characteristics

  • Typical duration: 10-30 seconds for a complete flow
  • Bottlenecks: API fetch on first run (~2-5 seconds)
  • Scaling considerations: Pagination handles large provider/model lists efficiently

Debugging This Flow

Key Log Messages

Message Meaning Action
Models database unavailable Both API and bundled file failed Check network, verify bundled file exists
No providers available Registry loaded but empty Check bundled JSON integrity
Cannot add model from {provider} Unsupported provider Choose different provider
{model} does NOT support tool calling Limited functionality model Confirm or cancel
Added {model_key} to extra_models.json Success Model ready to use

Common Issues

Model not appearing in picker after adding

Likely Cause: Config not reloaded Fix: Run /reload command or restart Code Puppy

API key not being recognized

Likely Cause: Environment variable not set correctly Fix: Verify with /get {ENV_VAR} or check ~/.code_puppy/config.json

Custom model not working

Likely Cause: Wrong model ID or context size Fix: Remove from extra_models.json and re-add with correct values
