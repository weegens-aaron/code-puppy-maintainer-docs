Flow: Tool Call Execution

What Happens?

When an agent decides to use a tool (like reading a file, running a command, or controlling a browser), the tool call goes through a multi-stage pipeline: registration → invocation → pre-hooks → execution → post-hooks → result handling. This flow ensures plugins can intercept, audit, and modify tool behavior while maintaining thread safety and cancellation support.

Trigger

  • Agent Decision: The LLM decides to call a tool based on the task and available tools
  • Parallel Execution: Multiple tools can be called simultaneously (parallel tool calls)
  • Sequential Execution: Tools execute one-at-a-time when dependencies exist

Outcome

  • Tool function executes with provided arguments
  • Result is returned to the agent for further reasoning
  • Plugins are notified before and after execution
  • Execution time and results are tracked for observability

Step-by-Step

flowchart TD
    A[Agent Requests Tool Call] --> B[Tool Registration Check]
    B -->|Known Tool| C[Validate Arguments]
    B -->|Unknown Tool| Z[Skip with Warning]
    C --> D[Fire pre_tool_call Hook]
    D --> E{Plugin Blocks?}
    E -->|Yes| F[Return Block Response]
    E -->|No| G[Execute Tool Function]
    G --> H[Track Execution Time]
    H --> I[Fire post_tool_call Hook]
    I --> J[Return Result to Agent]
    J --> K[Agent Continues Reasoning]

1. Tool Registration

  • What: Tools are registered when the agent is created via register_tools_for_agent()
  • Where: code_puppy/tools/__init__.py → register_tools_for_agent()
  • Failure mode: Unknown tools are skipped with a warning instead of crashing

Tools come from three sources:

  1. Core tools: Built-in tools in code_puppy/tools/ (file ops, browser control, etc.)
  2. Plugin tools: Registered via register_tools callback in plugins
  3. MCP tools: From Model Context Protocol servers

2. Argument Validation

  • What: Pydantic-ai validates tool arguments against the tool's schema
  • Where: Handled by pydantic-ai framework during tool call parsing
  • Failure mode: Invalid arguments cause the tool call to fail, agent retries

3. Pre-Tool Hook (pre_tool_call)

  • What: Fire async callbacks allowing plugins to inspect/modify/block tool calls
  • Where: code_puppy/callbacks.py → on_pre_tool_call()
  • Failure mode: Hook failures are logged but don't block execution

Plugins can:

  • Log tool calls for auditing
  • Block dangerous operations (via shell_safety plugin)
  • Modify arguments before execution

4. Tool Execution

  • What: The actual tool function runs with validated arguments
  • Where: Tool-specific handler (e.g., register_read_file → read_file())
  • Failure mode: Exceptions are caught and returned as error strings to the agent

Example tool structure:

async def read_file(ctx: RunContext, path: str, start_line: int | None = None) -> ReadFileOutput:
    try:
        # Tool logic here
        content = await read_file_async(path)
        return ReadFileOutput(content=content, num_tokens=estimate_tokens(content))
    except Exception as e:
        return ReadFileOutput(content=None, error=str(e))

5. Execution Time Tracking

  • What: Track how long the tool took to execute (in milliseconds)
  • Where: Calculated in the event stream handler and callback engine
  • Purpose: Observability and performance monitoring

6. Post-Tool Hook (post_tool_call)

  • What: Fire async callbacks with tool results for logging/analysis
  • Where: code_puppy/callbacks.py → on_post_tool_call()
  • Failure mode: Hook failures are logged but don't affect the result

Plugins can:

  • Log execution metrics
  • Cache results
  • Trigger side-effects based on tool outcomes

7. Result Handling

  • What: Tool result is formatted and returned to the agent
  • Where: Pydantic-ai framework converts result to ToolReturnPart
  • Failure mode: Errors are returned as strings, agent can retry or adapt

Data Transformations

Stage Input Output Location
Tool Request ToolCallPart(tool_name, args_json) Parsed arguments Pydantic-ai
Pre-Hook (tool_name, args_dict, context) None or modified args callbacks.py
Execution Validated arguments Tool-specific result (Pydantic model) Tool handler
Post-Hook (tool_name, args, result, duration_ms) None callbacks.py
Agent Response Result object ToolReturnPart(result) Pydantic-ai

Failure Handling

Failure Point Detection Recovery Alert?
Unknown tool Tool name not in registry Skip with warning No
Invalid args Pydantic validation error Agent retries No
Tool exception Exception in tool function Return error string No
Pre-hook block Plugin returns block signal Return block response Yes (logged)
Post-hook error Exception in callback Log error, continue No

Performance Characteristics

  • Typical duration: 10ms-2s depending on tool (file I/O vs. network calls)
  • Bottlenecks:
  • Disk I/O for file operations
  • Network latency for API calls
  • Browser automation for DOM operations
  • Scaling considerations: Parallel tool calls can overwhelm resources if not throttled

Debugging This Flow

Key Log Messages

Message Meaning Action
Registered async callback {name} for phase 'pre_tool_call' Plugin registered pre-hook Verify plugin loaded
Unknown tool '{name}' requested, skipping Agent requested non-existent tool Check agent config
Callback {name} failed in phase 'pre_tool_call' Hook threw exception Check plugin code
