Flow: Agent Execution

What Happens?

This flow describes how Code Puppy executes an agent task end-to-end - from receiving a user prompt to returning the final response. It encompasses model loading, tool registration, message history management, tool call execution, and streaming response handling.

Trigger

  • User action: Typing a prompt in the CLI and pressing Enter
  • API call: POST /api/commands/execute with agent name and prompt
  • Subagent invocation: Parent agent requests a subagent for specialized work

Outcome

  • Agent processes the prompt using the configured LLM model
  • Tools are called as needed to accomplish the task
  • Response is streamed back to the user in real-time
  • Message history is updated for future context
  • MCP tool definitions are cached for token estimation

Step-by-Step

flowchart TD
    A[User Prompt] --> B[Load/Reuse Agent]
    B --> C[Prepare Prompt with System Instructions]
    C --> D[Execute Agent Run]
    D --> E{Tool Call?}
    E -->|Yes| F[Execute Tool]
    F --> G[Return Tool Result]
    G --> D
    E -->|No| H[Stream Response]
    H --> I[Update Message History]
    I --> J{Context Limit?}
    J -->|Yes| K[Compaction/Summarization]
    J -->|No| L[Cache MCP Tools]
    K --> L
    L --> M[Return Final Response]

1. Agent Initialization

  • What: Load or reuse existing agent instance with configured model
  • Where: code_puppy/agents/base_agent.py:reload_code_generation_agent()
  • Failure mode: If model loading fails, falls back to global default model; if all models fail, raises ValueError

2. Prompt Preparation

  • What: Combine user prompt with system instructions, puppy rules, and attachments
  • Where: code_puppy/agents/base_agent.py:run_with_mcp()
  • Key steps:
  • Load system prompt from get_system_prompt()
  • Append agent identity (agent-name-uuid)
  • Load AGENT.md rules from global config and project directory
  • Apply model-specific transformations via prepare_prompt_for_model() hook
  • Handle attachments (images, documents) if present
  • Failure mode: Invalid Unicode in prompt is sanitized with surrogate replacement

3. Agent Run Execution

  • What: Execute the pydantic-ai agent with the prepared prompt
  • Where: code_puppy/agents/base_agent.py:run_with_mcp() → pydantic_agent.run()
  • Key components:
  • DBOS wrapper: If enabled, wraps execution in DBOS workflow for durability
  • Session ID: Unique workflow ID for tracking and cancellation
  • Usage limits: Configured message limit to prevent runaway runs
  • Event stream handler: Real-time response streaming via event_stream_handler
  • Failure mode: CancelledError on user interrupt, UsageLimitExceeded on message limit

4. Tool Call Handling

  • What: When agent requests a tool call, execute it and return result
  • Where: code_puppy/tools/ (various tool handlers)
  • Flow: 1. Agent returns ToolCallPart with tool name and arguments 2. Pydantic-ai dispatches to registered tool function 3. Tool executes (e.g., read_file, run_shell_command) 4. Result wrapped in ToolReturnPart and sent back to agent 5. Agent continues processing with tool result
  • Hooks: pre_tool_call and post_tool_call callbacks fire around each tool execution
  • Failure mode: Tool errors are returned as error content; agent may retry or handle gracefully

5. Message History Accumulation

  • What: Track conversation history for context in subsequent turns
  • Where: code_puppy/agents/base_agent.py:message_history_accumulator()
  • Process: 1. Hash each incoming message for deduplication 2. Add new messages not already in history 3. Always preserve last message (user prompt) even if hash matches 4. Trigger compaction if token limit exceeded
  • Failure mode: If history ends with ModelResponse instead of ModelRequest, trimmed to prevent API errors

6. Context Compaction

  • What: Manage token usage when history grows too large
  • Where: code_puppy/agents/base_agent.py:message_history_processor()
  • Strategies:
  • Summarization (default): Use summarization agent to compact old messages
  • Truncation: Drop old messages beyond protected token limit
  • Protection: Checks for pending tool calls before compaction; defers if tools pending
  • Failure mode: If summarization fails, falls back to original messages

7. Delayed Compaction

  • What: Handle compaction requests that were deferred due to pending tool calls
  • Where: code_puppy/agents/base_agent.py:run_with_mcp()
  • Trigger: At start of new run, checks should_attempt_delayed_compaction()
  • Process: If tool calls completed since last check, run compaction before processing new prompt
  • Failure mode: If compaction fails, continues with original history

8. Response Streaming

  • What: Stream agent response back to user in real-time
  • Where: code_puppy/agents/event_stream_handler.py
  • Event types:
  • content: Text content being generated
  • tool_call: Tool execution started
  • tool_result: Tool execution completed
  • usage: Token usage updates
  • Display: Rich renderer formats events for terminal output

9. MCP Tool Cache Update

  • What: After successful run, fetch MCP tool definitions for token estimation
  • Where: code_puppy/agents/base_agent.py:_update_mcp_tool_cache()
  • Why: Token estimation needs accurate tool definition sizes
  • Failure mode: If server unavailable, uses cached values from previous run

10. Cleanup & Hooks

  • What: Finalize run and fire completion hooks
  • Where: code_puppy/agents/base_agent.py:run_with_mcp() finally block
  • Hooks fired:
  • agent_run_end: Plugins notified of completion (success/error/response)
  • Signal handlers restored to original
  • Keyboard listener stopped if active
  • Failure mode: Hook errors are caught and logged; don't fail the run

Failure Handling

Failure Point Detection Recovery Alert?
Model loading fails ValueError from ModelFactory.get_model() Try fallback models; raise if all fail Yes (warning)
Tool execution fails Exception in tool handler Return error content to agent Yes (error)
Summarization fails SummarizationError Return original messages; try truncation Yes (warning)
User cancels CancelledError/KeyboardInterrupt Stop agent task; clean up resources No (info)
MCP server error McpError from tool call Return error to agent; suggest disabling Yes (info)
Context limit exceeded UsageLimitExceeded Inform user; allow "continue" Yes (info)

Performance Characteristics

  • Typical duration: 5-30 seconds for simple tasks; minutes for complex workflows
  • Bottlenecks:
  • LLM response time (depends on model/provider)
  • Tool execution (file I/O, shell commands, network requests)
  • Summarization (additional LLM call for compaction)
  • Scaling considerations:
  • DBOS mode adds durability but may have overhead
  • Large message history increases compaction time
  • Many MCP servers increase tool definition loading time

Common Issues

Agent hangs on tool call

Likely Causes:

  1. Tool is waiting for user input (e.g., file permission prompt)
  2. Shell command is blocking
  3. MCP server is unresponsive

Diagnosis:

# Check if shell command is running
# Look for "press Ctrl+X to cancel" message

Resolution:

  • Press Ctrl+X to cancel shell commands
  • Press configured cancel key (default Ctrl+C) to cancel agent

Context limit errors

Likely Causes:

  1. Long-running conversation without compaction
  2. Compaction threshold set too high
  3. Protected tokens too large

Resolution:

  1. Check config get compaction_threshold (default: 0.8)
  2. Check config get protected_token_count (default: 40000)
  3. Use /reset to start fresh conversation

Model not found errors

Likely Causes:

  1. Agent has pinned model that doesn't exist
  2. Global model not configured

Resolution:

  1. Check config get models for available models
  2. Use /pin_model to set valid model for agent
  3. Use config set model=<name> to set global default
