Concept: Callback System

What Is This?

The Callback System is Code Puppy's plugin extension mechanism that allows plugins to hook into various phases of the application lifecycle. It uses a publish-subscribe pattern where plugins register callback functions that are triggered at specific events, enabling modular, non-invasive extensions to core functionality.

Why This Approach?

The callback system uses event-driven architecture to:

  • Enable modularity: Plugins can extend without modifying core code
  • Support composition: Multiple plugins can react to the same event
  • Allow interception: Plugins can modify or block operations
  • Provide observability: Track and log system events
  • Maintain separation: Core and plugins remain decoupled

How It Works

The Pattern

sequenceDiagram
    participant Core
    participant Dispatcher
    participant Plugin1
    participant Plugin2
    
    Core->>Dispatcher: Trigger event
    Dispatcher->>Plugin1: Call callback
    Plugin1-->>Dispatcher: Return result
    Dispatcher->>Plugin2: Call callback
    Plugin2-->>Dispatcher: Return result
    Dispatcher-->>Core: Aggregate results

Example

from code_puppy.callbacks import register_callback

# Register a callback for startup
def on_app_startup():
    print("Plugin initialized!")

register_callback("startup", on_app_startup)

# Register a callback for tool calls
async def on_tool_call(tool_name, tool_args, context=None):
    print(f"Tool {tool_name} called with {tool_args}")

register_callback("pre_tool_call", on_tool_call)

Registration

Basic Registration

from code_puppy.callbacks import register_callback

def my_callback():
    print("Event occurred!")

register_callback("startup", my_callback)

Async Callbacks

import asyncio

async def async_callback():
    await asyncio.sleep(1)
    print("Async event handled!")

register_callback("startup", async_callback)

Plugin Registration

# In plugin's register_callbacks.py
from code_puppy.callbacks import register_callback

def _on_startup():
    print("Plugin loaded!")

def _on_tool_call(tool_name, tool_args, context=None):
    print(f"Tool called: {tool_name}")

register_callback("startup", _on_startup)
register_callback("pre_tool_call", _on_tool_call)

Triggering Callbacks

Sync Triggers

from code_puppy.callbacks import on_load_prompt

# Returns list of results from all callbacks
results = on_load_prompt()

Async Triggers

from code_puppy.callbacks import on_stream_event

# Returns coroutine
async def handle_stream():
    results = await on_stream_event("token", {"text": "hello"})

Blocking vs Non-Blocking

  • Blocking hooks can return True/False to allow/deny operation
  • Non-blocking hooks are for observation only
  • All callbacks are called even if one fails

Error Handling

Callback Failures

# Callbacks that fail are logged but don't break the flow
try:
    result = callback(*args, **kwargs)
except Exception as e:
    logger.error(f"Callback {callback.__name__} failed: {e}")
    results.append(None)

Duplicate Prevention

# Same callback can't be registered twice
if func in _callbacks[phase]:
    logger.debug(f"Callback {func.__name__} already registered, skipping")
    return

Return Values

Blocking Hooks

# Can block operations
def on_file_permission(context, file_path, operation, ...):
    # Return False to deny
    if not is_allowed(file_path):
        return False
    return True

# Can block shell commands
def on_run_shell_command(context, command, ...):
    if is_blocked(command):
        return {"blocked": True}
    return None

Data Collection Hooks

# Collect data from multiple plugins
def on_register_tools():
    # Returns list of tool definitions
    return [
        {"name": "tool1", "register_func": register_tool1},
        {"name": "tool2", "register_func": register_tool2},
    ]

Custom Commands

# Handle custom slash commands
def on_custom_command(command, name):
    if name == "mycommand":
        # Return True to indicate handled
        return True
    # Return None to continue to next handler
    return None

Conventions & Rules

IMPORTANT

One registration per plugin: Each plugin should have one register_callbacks.py

IMPORTANT

Fail gracefully: Never let callback errors crash the app

IMPORTANT

Return None for non-handlers: Return None from hooks you don't own

IMPORTANT

Use async for I/O: Use async callbacks for network/disk operations

Anti-Patterns to Avoid

CAUTION

Don't block unnecessarily: Only block when you need to prevent an operation

CAUTION

Don't raise exceptions: Log errors instead of raising

CAUTION

Don't modify shared state: Use thread-safe operations

CAUTION

Don't register duplicates: The system prevents this, but it's wasteful

Extending This

To add a new callback phase:

  1. Add to PhaseType in callbacks.py: ```python PhaseType = Literal[ # ... existing phases ... "my_new_phase", ] ```
  1. Initialize the list: ```python _callbacks: Dict[PhaseType, List[CallbackFunc]] = { # ... existing phases ... "my_new_phase": [], } ```
  1. Create trigger function: ```python async def on_my_new_phase(args, *kwargs): return await _trigger_callbacks("my_new_phase", args, *kwargs) ```
  1. Document the phase in this concept doc
