Concept: Error Handling Strategy

What Is This?

Error Handling Strategy defines how Code Puppy detects, logs, recovers from, and communicates errors throughout the system. It uses a multi-layered approach with structured logging, error isolation, and user-friendly messages to maintain system stability while providing actionable diagnostics.

Why This Approach?

The error handling system uses defensive programming with graceful degradation to:

  • Maintain stability: One error doesn't crash the entire system
  • Enable debugging: Detailed logs for troubleshooting
  • Protect users: Clear error messages without exposing internals
  • Support recovery: Automatic retry and fallback mechanisms
  • Track issues: Centralized error logging for analysis

Example

from code_puppy.error_logging import log_error

try:
    result = risky_operation()
except Exception as e:
    # Log with context
    log_error(e, context="Failed to process file")
    
    # Provide fallback
    result = fallback_value()
    
    # Notify user (via messaging system)
    emit_warning(f"Could not process file: {e}")

Circuit Breaker Pattern

from code_puppy.mcp_.circuit_breaker import CircuitBreaker

breaker = CircuitBreaker(
    failure_threshold=5,
    recovery_timeout=30,
    half_open_requests=3
)

try:
    with breaker:
        result = risky_operation()
except CircuitOpenError:
    # Circuit is open, operation blocked
    result = fallback_value()

States:

  • Closed: Normal operation
  • Open: Too many failures, block requests
  • Half-Open: Testing if service recovered

def get_config_value(key, default=None):
    try:
        return get_value(key)
    except Exception as e:
        log_error(e, context=f"Failed to get config: {key}")
        return default

Conventions & Rules

IMPORTANT

Never crash: Catch exceptions at boundaries, don't let them propagate

IMPORTANT

Log everything: All errors must be logged with context

IMPORTANT

Fail gracefully: Provide fallbacks when possible

IMPORTANT

Don't leak: Don't expose internal errors to users

Anti-Patterns to Avoid

CAUTION

Don't swallow errors: Always log or handle exceptions

CAUTION

Don't block: Logging should never block main execution

CAUTION

Don't expose: Don't show stack traces to end users

CAUTION

Don't ignore: Silent failures make debugging impossible

Error Handling by Component

Agents

from code_puppy.error_logging import log_error

try:
    result = await agent.run(prompt)
except Exception as e:
    log_error(e, context=f"Agent {agent.name} failed")
    return "Sorry, I encountered an error while processing your request."

Tools

def read_file(file_path):
    try:
        with open(file_path) as f:
            return f.read()
    except FileNotFoundError:
        emit_error(f"File not found: {file_path}")
        return None
    except Exception as e:
        log_error(e, context=f"Failed to read file: {file_path}")
        emit_error(f"Error reading file: {e}")
        return None

MCP Servers

async def connect_server(config):
    try:
        server = await create_server(config)
        await server.start()
        return server
    except Exception as e:
        log_error(e, context=f"Failed to start server: {config.name}")
        isolator.record_error(config.id, e)
        raise

tail -f ~/.code_puppy/logs/errors.log

# Search for specific error
grep "FileNotFoundError" ~/.code_puppy/logs/errors.log

import logging
logging.getLogger("code_puppy").setLevel(logging.DEBUG)

Reproduce Issues

  1. Check error logs for context
  2. Reproduce the action that caused the error
  3. Check component state (MCP servers, agents, etc.)
  4. Review recent changes to config or code
